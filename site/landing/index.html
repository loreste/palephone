Install Pale Server on Linux

The supported Linux path in this repository is Docker Compose. It runs Pale Server with PostgreSQL, coturn, and NATS, then exposes the HTTP API on port 8090 and SIP on 5060/5061.

  1. Install Docker and Docker Compose on a Linux host.
  2. Clone the repository and enter the project directory.
  3. Generate secrets, review the generated .env, then start the stack.
git clone https://github.com/loreste/palephone.git
cd palephone
./scripts/generate-secrets.sh
docker compose up -d
curl http://localhost:8090/health

Important Linux settings

  • PALE_SERVER_TOKEN, PALE_ADMIN_PASSWORD, and PALE_STORAGE_KEY are required and must be at least 24 characters.
  • PALE_DATABASE_URL points Pale Server at PostgreSQL. Docker Compose sets this for you.
  • PALE_SIP_EXTERNAL_ADDR should be passed to Pale Server for remote SIP clients, especially when using a registrar path.
  • PALE_SIP_BACKEND defaults to pjsip. The udp-parser backend is a development fallback for the current built-in REGISTER/PBX parser path and also requires PALE_ALLOW_INSECURE_SIP_UDP=1.
For public deployments, put TLS in front of the HTTP API or set PALE_HTTP_TLS_CERT and PALE_HTTP_TLS_KEY. Open only the ports your deployment actually uses.

Install Pale Server on Windows

Windows can run Pale Server as a Rust-built executable. The production container path is Linux-first today, so treat the Windows install as a direct server runtime unless you wrap it with your own service manager such as WinSW or NSSM.

  1. Install Visual Studio Build Tools with the C++ workload.
  2. Install Rust for the MSVC toolchain.
  3. Install PostgreSQL if you do not want local SQLite fallback.
  4. Build the server binary from the repository.
git clone https://github.com/loreste/palephone.git
cd palephone\src-tauri\crates\pale-server
cargo build --release --bin pale-server

Set the required environment variables

$env:PALE_SERVER_TOKEN = "replace-with-strong-token-24-plus-chars"
$env:PALE_ADMIN_PASSWORD = "replace-with-strong-admin-password"
$env:PALE_STORAGE_KEY = "replace-with-strong-storage-key"
$env:PALE_HTTP_ADDR = "0.0.0.0:8080"
$env:PALE_DATABASE_URL = "host=localhost user=pale password=... dbname=pale"
..\..\target\release\pale-server.exe

Verify with curl http://localhost:8080/health. If Windows Firewall is enabled, allow the HTTP API and any SIP/TURN ports you plan to expose.

How-tos

Create an admin user

Pale Server ships with an admin account controlled by PALE_ADMIN_USERNAME and PALE_ADMIN_PASSWORD. You can also create users through the server CLI. 

docker compose exec pale-server pale-server user create \
  --username sip:admin@example.com \
  --display-name "Admin" \
  --password "replace-with-a-strong-password" \
  --role admin

Point a client at your server

  1. Install the Palephone client for macOS, Linux, Windows, or Android.
  2. Open Settings, set the server URL, and sign in with the SIP URI and password for that user.
  3. Admin appears only for users whose server role is admin.

Check server health

curl https://your-domain.example/health

A healthy server returns JSON with ok, service, and status fields.

TLS guidance

We recommend TLS anywhere Pale is reachable outside a trusted local machine. The server handles sign-in, bearer tokens, admin actions, files, call records, presence, and meeting data. Sending that over plain HTTP makes credentials and session traffic easier to intercept.

HTTP API

The simplest production pattern is to terminate HTTPS at a reverse proxy such as Nginx, Caddy, or Traefik, then proxy to Pale Server on localhost or a private network. Pale Server can also serve TLS directly when PALE_HTTP_TLS_CERT and PALE_HTTP_TLS_KEY are set. 

PALE_HTTP_ADDR=0.0.0.0:8080
PALE_HTTP_TLS_CERT=/etc/letsencrypt/live/your-domain/fullchain.pem
PALE_HTTP_TLS_KEY=/etc/letsencrypt/live/your-domain/privkey.pem

SIP signaling

SIP TLS should be used when SIP traffic crosses networks you do not fully control. Set PALE_SIP_TLS_CERT and PALE_SIP_TLS_KEY to enable SIP TLS, and expose port 5061 only when your deployment actually uses it. 

PALE_SIP_TLS=true
PALE_SIP_TLS_CERT=/etc/letsencrypt/live/your-domain/fullchain.pem
PALE_SIP_TLS_KEY=/etc/letsencrypt/live/your-domain/privkey.pem
PALE_SIP_TLS_PORT=5061
TLS protects signaling and API traffic. Media privacy still depends on your media path, SRTP settings, and TURN deployment, so test calls from the real networks your users will use.

Troubleshooting

Sign-in says unauthorized

  • Confirm the client is using the same server URL you tested with /health.
  • Check the SIP URI format. Use sip:name@domain when the user was created that way.
  • Reset the user's password from the admin UI or recreate the user with the CLI.

The Admin tab disappears

  • The tab is role-based. The signed-in user must have role admin.
  • Sign out and sign back in after changing a user's role so the client refreshes the stored session.

SIP clients do not register

  • Pass PALE_SIP_EXTERNAL_ADDR to Pale Server with the public SIP host and port clients should use.
  • Use a dedicated SIP registrar/proxy for production, or use PALE_SIP_BACKEND=udp-parser plus PALE_ALLOW_INSECURE_SIP_UDP=1 only when you intentionally accept the current fallback parser path.
  • Check firewall rules for SIP and RTP/TURN traffic.

Calls connect but audio fails

  • Verify TURN settings: PALE_TURN_SERVER, PALE_TURN_USERNAME, PALE_TURN_PASSWORD, and PALE_TURN_REALM.
  • Make sure UDP relay ports are reachable from the client networks.
  • If you terminate TLS at a proxy, confirm WebSocket/SSE and long-running HTTP requests are not being buffered or dropped.

Browsers or clients reject the server

  • Check that the certificate covers the exact hostname users enter in the client.
  • Confirm the full certificate chain is served, not just the leaf certificate.
  • If TLS terminates at a proxy, forward the original host and protocol headers so generated URLs stay correct.

Production notes

Pale gives you the communications surface and the admin hooks, but a production deployment still needs real infrastructure around it: DNS, TLS certificates, backups, monitoring, a TURN relay, PostgreSQL, and any external providers you decide to use for AI, speech, malware scanning, storage, PSTN, or E911.

  • Keep .env and generated secrets out of Git.
  • Back up PostgreSQL and any file storage location before upgrades.
  • Rotate PALE_SERVER_TOKEN and admin passwords when staff changes.
  • Use the admin readiness reports to separate configured providers from fully tested integrations.
