## 1. SIP stack: in-dialog relay foundation (top priority)

The udp-parser backend (`pale-server/src/sip.rs`) has NO in-dialog request
forwarding: INFO (DTMF), BYE, REFER, and re-INVITE are absorbed locally.

## proxy_invite relays only the FIRST response, then drops the transaction — calls through the proxy path can never complete
Severity: medium
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:928-951
Evidence: let mut response = vec![0_u8; 16 * 1024];
let (len, _) = match timeout(StdDuration::from_secs(5), proxy_socket.recv_from(&mut response)).await { ... };
let relayed = strip_proxy_via(&response, &branch).unwrap_or(response);
... Some(relayed)
— exactly one recv_from on an ephemeral socket; the socket is dropped when the function returns. A UAS that sends 100 Trying or 180 Ringing first (Zoiper, Linphone, Bria all do) gets only that provisional relayed; the subsequent 200 OK arrives at a dead socket and is lost. Also `if status >= 200 { SipDialogStatus::Ended }` marks the dialog Ended on a 200 OK (answered != ended). ACK is never forwarded (handle_ack at sip.rs:471 only mutates local state).
Fix design: Make proxy_invite loop on the ephemeral socket: forward every provisional (status < 200) back to the original `peer` immediately via the server socket (send_to(stripped, peer)), and only return when a final response (>=200) arrives or an overall timer (e.g. 32 s, Timer C-ish) expires. After relaying a final non-2xx, also synthesize and send the hop-by-hop ACK to the UAS from the proxy socket (RFC 3261 17.1.1.3) so the UAS stops retransmitting. For 2xx, keep the proxy socket alive in a spawned task for ~32 s to absorb 200 retransmissions and re-relay them to `peer` (end-to-end ACK goes direct because no Record-Route is inserted — acceptable if you stay out of the dialog path, but then see the in-dialog-forwarding finding). On 2xx set SipDialogStatus::Answered/Confirmed (add the variant if missing), Ended only on BYE/CANCEL/non-2xx final. handle_udp_packet (sip.rs:62) must pass `peer` into proxy_invite so provisionals can be relayed mid-flight.
Verifier notes on the fix: Implementable as designed: handle_udp_packet already has `peer`, server_socket is already a parameter of proxy_invite, and each packet runs in its own tokio task (sip.rs:39) so a ~32 s Timer-C loop is harmless. Relaying stripped provisionals from the server socket (same ip:port the caller signaled to) is correct for NAT. Hop-by-hop ACK for non-2xx finals and a spawned ~32 s task to absorb/relay 2xx retransmissions are standard and constructible from in-scope data; adding an Answered/Confirmed variant to SipDialogStatus is needed (enum currently lacks it). Two adjacent gaps the fix leaves open (pre-existing, acceptably deferred): no transaction matching, so caller INVITE retransmissions before the first provisional spawn duplicate upstream transactions with fresh branches; and CANCEL is not forwarded upstream (handle_cancel at sip.rs:507 answers 200 locally), so abandoning during the new longer window leaves the UAS ringing — should be handled by the related in-dialog-forwarding finding.

## Locally generated responses never add a To-tag, and 200 OK to INVITE has no Contact and no SDP answer — server-terminated calls (queue hold, IVR, voicemail, re-INVITE) are unusable
Severity: medium
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:1126-1145
Evidence: fn response(&self, ...) { for header in ["via", "from", "to", "call-id", "cseq"] { if let Some(value) = self.header(header) { headers.push(response_header(header, value.to_string())); } } ... body: Vec::new() }
— To is echoed verbatim with no tag (violates RFC 3261 §8.2.6.2: UAS MUST add a to-tag to non-100 responses). Every 200 OK to INVITE (sip.rs:181 re-INVITE, :257 conference, :321 queue, :378 IVR, :388/:432/:452 voicemail) has Content-Length: 0 (no SDP answer, violates offer/answer §13.2.1) and no Contact (violates §12.1.1: 2xx to INVITE MUST contain Contact). Standard clients cannot construct the dialog (no remote tag, no remote target) and will fail the call; the ACK they generate is also unmatchable.
Fix design: In response(): when echoing To on a non-100 response to a non-ACK request, append `;tag=<stable tag>` if the request's To has none. Derive the tag deterministically so retransmitted requests get the same tag: e.g. first 16 hex of md5_hex(call_id + from-tag + cseq-method). For the 2xx-to-INVITE paths, add ("Contact", format!("<sip:pale@{}>", advertised_sip_addr)) and an SDP answer. Since the server has no media engine on this backend, the honest fix is: stop answering INVITE with bodyless 200 OK — return 480/486 (or implement a minimal SDP answer with a recvonly audio stream bound to a real RTP port if hold-music is intended). For the re-INVITE path (sip.rs:181), echo back the offered SDP with a=recvonly/a=inactive as an answer, Content-Type: application/sdp, correct Content-Length.
Verifier notes on the fix: Adding a deterministic to-tag in response() (skip 100, skip when request To already has a tag) and a Contact header on 2xx-to-INVITE is straightforward and correct; the md5-derived stable tag handles retransmissions. Replacing bodyless 200 OK with 480/486 on the no-media paths is implementable but changes product/CDR semantics (answered/voicemail become rejections) — a deliberate product decision, defensible since those features had no media anyway. One flaw: for the re-INVITE path, echoing the client's offered SDP back as the answer is not a valid SDP answer — the c=/m= lines carry the client's own address/ports, directing the client to send media to itself; with a=inactive no media flows so it may pass in practice, but a correct fix must rewrite origin/connection lines or build a minimal answer. Also note the re-INVITE branch is largely unreachable in practice since the server never Record-Routes, so in-dialog requests rarely arrive at it.

## Feature gap (a): no in-dialog request forwarding exists at all — INFO (DTMF) is swallowed with a local 200 OK and never reaches the peer leg
Severity: high
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:522-530
Evidence: fn handle_info(request: &SipRequest, state: &AppState) -> Option<String> { ... if !request.is_authorized(state, &realm) { return Some(request.digest_challenge(...)); } Some(request.response(200, "OK", &[])) }
— RFC 6086 §4.2.2 requires a UA that doesn't understand the Info Package to reject (469/415), and a proxy to relay; blindly 200 OK'ing DTMF INFO makes the sender believe digits were delivered. Confirmed there is NO in-dialog forwarding mechanism anywhere: BYE (sip.rs:484), ACK (sip.rs:471) and INFO are all answered/absorbed locally; only initial INVITE (proxy_invite, sip.rs:874) and out-of-dialog MESSAGE (relay_message, sip.rs:803) are forwarded. SipDialog (lib.rs:2910-2920) stores only call_id/from_uri/to_uri/target_contact — no tags, no peer source addresses, no contacts — so the peer leg cannot currently be addressed.
Fix design: (1) Extend SipDialog (lib.rs:2910) with from_contact: Option<String>, from_source: Option<String>, to_source: Option<String>; populate in proxy_invite's upsert (sip.rs:912-919): from_contact = request.header("contact"), from_source = peer.to_string() (thread `peer` into proxy_invite from handle_udp_packet), to_source = target_addr.to_string(). Add `pub fn dialog_for(&self, call_id: &str) -> Option<SipDialog>` next to dialog_exists (lib.rs:1128). (2) New async fn relay_in_dialog_request(server_socket, packet, request, peer, state): look up dialog by Call-ID; pick the peer leg by comparing the request's From URI to dialog.from_uri (match → target = target_contact/to_source, else → from_contact/from_source); resolve addr via stored source first, falling back to invite_target() (sip.rs:954). (3) Generalize rewrite_invite_for_proxy (sip.rs:975-1001): remove the `if method != "INVITE" { return None; }` guard at :987, keep the original Via chain, push a fresh `Via: ...;branch=z9hG4bK-pale-{uuid}` on top, decrement Max-Forwards while copying lines, keep R-URI = target contact. (4) Mirror proxy_invite's ephemeral-socket send/recv/strip_proxy_via to relay the final response back; 408 on timeout. (5) Hook into handle_udp_packet after the MESSAGE block (sip.rs:72): `if matches!(request.method().as_str(), "INFO" | "BYE" | "UPDATE") && request.call_id().map(|c| state.dialog_exists(c)).unwrap_or(false) { relay_in_dialog_request(...).await }` — keep the existing local bookkeeping (dialog status, CDR) but source the response from the peer leg.
Verifier notes on the fix: Design is implementable and consistent with existing proxy_invite patterns (ephemeral socket, single recv, strip_proxy_via; From-URI direction heuristic works since no tags are stored anyway). Three gaps: (1) It omits ACK forwarding — in the outbound-proxy topology where this fix matters, handle_ack still absorbs the ACK for the proxied 200 OK, so the callee retransmits and tears down the dialog per RFC 3261 §13.3.1.4; ACK must be added to the relayed method set (fire-and-forget, no response wait). (2) Authorization handling is unspecified: the local leg should still be digest-checked, but the remote/callee leg cannot answer a local-realm challenge, so the relay must not reuse is_authorized blindly for inbound peer-leg requests. (3) Single 5s recv misses provisional-then-final sequences, though that is rare for non-INVITE transactions and matches the existing proxy_invite limitation. Alternatively, the minimal RFC 6086-correct fix for INFO alone is returning 469 Bad Info Package / 415 instead of 200; the proposed relay is the more useful behavior. CANCEL relay remains unaddressed but is architecturally out of reach with ephemeral per-transaction sockets.

## Feature gap (b): REFER returns 202 and stops — no NOTIFY/sipfrag implicit subscription (RFC 3515 MUST), no transfer execution, and an illegal Subscription-State header on the 202
Severity: medium
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:556-664
Evidence: Some(request.response(202, "Accepted", &[("Subscription-State", "terminated;reason=noresource".to_string())]))
— handle_refer marks dialogs Ended (sip.rs:642-649) and records an audit event, but never NOTIFYs the referrer (RFC 3515 §2.4.4: the agent MUST create an implicit subscription and send an immediate NOTIFY with message/sipfrag) and never contacts the Refer-To target or the dialog peer. Subscription-State is only defined for NOTIFY, not for 202 (RFC 6665 §7.2); putting it on the 202 does not legally terminate the subscription — that requires Refer-Sub: false per RFC 7647, which is not checked. Transferring clients (Zoiper/Linphone/Bria) wait for the NOTIFY sipfrag before tearing down the call and will report transfer failure.
Fix design: Move REFER into the async path: in handle_udp_packet add a branch beside MESSAGE (sip.rs:67) calling a new async handle_refer_async(socket, request, peer, state). Minimal correct behavior: (1) honor RFC 7647 — if the REFER carries `Refer-Sub: false`, reply 202 with `Refer-Sub: false` and send no NOTIFY; (2) otherwise reply plain 202 (remove the Subscription-State tuple at sip.rs:589-593 and :659-663), then immediately send a NOTIFY on the server socket to `peer`: R-URI = REFER's Contact URI (fallback `sip:{peer}`); Via with fresh z9hG4bK branch and sip_sent_by(local_addr); From = REFER's To header, appending `;tag=<deterministic tag from finding 2>` if absent; To = REFER's From header verbatim; Call-ID = same; CSeq `1 NOTIFY`; `Event: refer;id=<numeric CSeq of the REFER>`; `Subscription-State: terminated;reason=noresource`; `Content-Type: message/sipfrag;version=2.0`; body `SIP/2.0 200 OK\r\n` — use 200 only after the transfer action actually succeeded, else `SIP/2.0 503 Service Unavailable\r\n`. (3) For proxied dialogs, instead of fabricating success, relay the REFER to the peer leg via relay_in_dialog_request from finding 3 and pass its real response code back; only generate the local terminal NOTIFY for dialogs the server itself terminates (park/pickup/queue). (4) Do not mark the dialog Ended on REFER alone (sip.rs:642) — blind-transfer dialogs end when the subsequent BYE arrives.
Verifier notes on the fix: The core of the fix is implementable as designed: the async branch pattern beside MESSAGE exists (sip.rs:67-72), relay_message (803-872) is a working template for sending requests on the server socket with sip_sent_by and fresh z9hG4bK branches, and a single immediate NOTIFY with Subscription-State: terminated carrying the final sipfrag is RFC-3515-permitted and accepted by real clients. The Refer-Sub handling per RFC 4488/7647 is correct. Caveats: (1) it depends on helpers from findings 2 (deterministic tag) and 3 (relay_in_dialog_request) that do not yet exist — viability is conditional on those landing; (2) part 3 is partly aspirational: proxy_invite does not Record-Route (rewrite_invite_for_proxy, 975-1001) and marks dialogs Ended on any final response (941-950), so in-dialog REFERs from the far UA never traverse this server and proxied-dialog relay needs more dialog-state work than the fix implies; (3) the sync handle_request path (pub handle_packet, used by all tests) must also be updated or REFER will behave differently per transport, and the test at 1949-1975 asserting Ended must be revised per fix item 4; (4) for plain blind transfers the honest sipfrag is 503 until relay exists — the fix makes failure protocol-clean and interoperable but does not by itself make transfers succeed.

## CANCEL is digest-challenged, violating RFC 3261 §22.1; no 487 for the INVITE; CANCEL during an in-flight proxied INVITE is never forwarded
Severity: high
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:507-520
Evidence: fn handle_cancel(...) { ... if !request.is_authorized(state, &realm) { return Some(request.digest_challenge(&realm, state.issue_sip_nonce())); } ... }
— RFC 3261 §22.1: "a UAS MUST NOT attempt to challenge a CANCEL" (UAs cannot supply credentials in CANCEL; any copied Authorization carries an already-consumed nonce given the single-use nonce store, so the challenge can never be satisfied → callers cannot abandon ringing calls). Matching is by Call-ID only instead of the top Via branch of the INVITE transaction (§9.2), no 487 Request Terminated is ever sent for the INVITE, and because proxy_invite blocks for up to 5 s with no shared transaction state, a CANCEL arriving meanwhile is not forwarded — the callee keeps ringing.
Fix design: Delete the is_authorized/digest_challenge block from handle_cancel; answer 200 OK immediately (hop-by-hop). Add a transaction map in AppState: ShardedMap<String /* top Via branch of received INVITE */, PendingInvite { call_id, target_addr, forwarded_branch, cancel_flag: Arc<AtomicBool> or a oneshot/Notify }>. proxy_invite registers itself on entry, removes itself on final response. handle_cancel matches the CANCEL's top Via branch against that map; if found, build `CANCEL <same R-URI as forwarded INVITE>` with the SAME Via branch as the forwarded INVITE (§9.1) and send it to target_addr, and have proxy_invite answer the original INVITE with 487 (which strip/relay logic already supports) once the UAS's 487 arrives or locally on timeout. Keep the existing dialog-status/CDR bookkeeping.
Verifier notes on the fix: Fix is RFC-correct and implementable, with one load-bearing gap to address explicitly: proxy_invite currently returns on the FIRST datagram received (typically a 100/180 provisional), so 'removes itself on final response' requires rewriting the receive path into a loop that relays provisional responses directly on server_socket to the caller and keeps the PendingInvite registered until a >=200 final response arrives. Without that, the map entry is gone after 100 Trying and CANCEL is still lost in the common case. Also consider: extending the 5s overall timeout (Timer C-like) or the cancellable window remains tiny; constructing the proxy CANCEL with same Call-ID/From/To/CSeq-number plus the forwarded branch (fix already covers branch and R-URI); and optionally gating unauthenticated CANCEL on source-address match with the INVITE's peer per §22.1 same-hop guidance, since handle_cancel mutates dialog status and CDRs.

## Single-use nonce is consumed by a relay path that can still fall through, causing unbreakable 401 loops; ACK authorization always fails
Severity: medium
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:803-820 and /Users/loreste/palephone/src-tauri/crates/pale-server/src/lib.rs:1026-1030
Evidence: relay_message: `if !request.is_authorized(state, &realm) { return None; }` (sip.rs:815, consumes the nonce via state.consume_sip_nonce, lib.rs:1029 `self.sip_nonces.remove(...)`) THEN `let (target_contact, target_addr) = invite_target(state, &to_uri)?;` (sip.rs:820). A correctly authenticated MESSAGE to an unregistered/unresolvable target consumes the nonce, returns None, falls into handle_message (sip.rs:95→532) which calls is_authorized again with the now-consumed nonce → 401 with a fresh nonce → client retries → loop forever; the message is never stored or rejected definitively. Similarly handle_ack (sip.rs:475) calls is_authorized, but an ACK's Authorization (copied from the INVITE per §22.2) carries the nonce the INVITE already consumed, so ACK auth always fails and the dialog-status update at sip.rs:479 never runs.
Fix design: Authorize exactly once per packet: in handle_udp_packet, perform the digest check up front and pass an `authorized: bool` (plus the verified username) into proxy_invite/relay_message/handle_request instead of each handler re-running is_authorized. Inside relay_message, move the invite_target lookup BEFORE the auth check (mirror proxy_invite's order at sip.rs:890-893) and, when the target is unresolvable, return a definitive 404/480 rather than None. For ACK, drop the auth requirement entirely (ACK is not challengeable). Additionally make nonce consumption transaction-scoped: store consumed nonces with the (branch, CSeq) that used them for ~64*T1 so retransmissions of the same request still verify, while a different request reusing the nonce is rejected with stale=true in the new challenge so well-behaved clients re-auth silently.
Verifier notes on the fix: The fix is implementable as designed: ACK auth removal is RFC 3261-correct (ACK is not challengeable); moving the invite_target lookup before nonce consumption mirrors proxy_invite (sip.rs:890-893); transaction-scoped nonce tracking with stale=true on reuse is the standard remedy for retransmissions. Three caveats: (a) the fix names only handle_udp_packet, but handle_packet (sip.rs:79) is a second entry point into the same handlers and must get the same authorized-flag treatment or the double-auth bug persists on that path; (b) a single up-front digest check must replicate per-method realm derivation (handle_register derives realm from To at sip.rs:111-118; other handlers use From/sender_realm); (c) returning a definitive 404/480 from relay_message kills handle_message's store-and-forward (sip.rs:543) for offline users — a smaller alternative is to return None from relay_message before consuming the nonce when the target is unresolvable, which breaks the loop AND restores offline message storage via handle_message. Also, keying consumed nonces by (branch, CSeq) is fine but branch alone (the transaction ID) is sufficient; moot for ACK once its auth requirement is dropped.

## Max-Forwards is never checked or decremented on any forwarded request; relay_message drops it entirely — proxy loops are unbounded
Severity: medium
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:975-1001 and 848-867
Evidence: rewrite_invite_for_proxy copies all original header lines verbatim (`for line in lines { out.push_str(line); }`) — Max-Forwards passes through undecremented (RFC 3261 §16.6 step 3 violation). relay_message builds a brand-new MESSAGE (sip.rs:848-867) whose header set is only Via/From/To/Call-ID/CSeq/Content-Type/Content-Length — Max-Forwards is absent (§8.1.1.6 says it SHOULD be present; a downstream proxy can't loop-bound it). grep confirms the string "max-forwards" appears nowhere in sip.rs. If a registration's contact/source resolves back to this server (spoofable via REGISTER), INVITEs self-forward; today the loop only breaks by accident because the consumed nonce fails re-auth.
Fix design: In rewrite_invite_for_proxy (and the generalized rewrite_request_for_proxy from finding 3): while copying lines, detect `max-forwards:` case-insensitively, parse the value; if 0, abort the forward and return 483 Too Many Hops to the sender; otherwise write `Max-Forwards: {n-1}`. If no Max-Forwards line existed, insert `Max-Forwards: 69`. In relay_message, add a `Max-Forwards: {orig-1, default 69}` header line to the constructed request. Also add cheap loop protection: if `target_addr` equals the server's own bound address(es), refuse with 482 Loop Detected.

## Dialog state machine is wrong: re-INVITE detection matches any prior Call-ID (even Ended dialogs), ACK sets status to Ringing, BYE records CDR disposition 'answered' unconditionally
Severity: medium
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:164-183, 471-482, 491-494
Evidence: handle_invite: `if state.dialog_exists(call_id) { ... return Some(request.response(200, "OK", &[])); }` — dialog_exists (lib.rs:1128) checks mere presence, so a NEW call that reuses a Call-ID seen before (including dialogs already in SipDialogStatus::Ended/Cancelled) is mis-treated as a re-INVITE and blindly 200-OK'd, bypassing all routing/auth-realm logic; there is no to-tag check to distinguish in-dialog from initial (RFC 3261 §12.2.2 matching requires Call-ID + tags). handle_ack: `let _ = state.update_sip_dialog_status(call_id, SipDialogStatus::Ringing);` — ACK confirms the dialog; setting Ringing moves the state machine backwards. handle_bye: `state.record_cdr_end(call_id, "answered")` even for calls that were never answered.
Fix design: Re-INVITE detection: require (a) the request's To header to carry a tag AND (b) the stored dialog status to be a live one (Ringing/Answered/Held), i.e. `state.dialog_for(call_id).map(|d| !matches!(d.status, Ended|Cancelled)).unwrap_or(false) && request.header("to").map(|t| t.contains("tag=")).unwrap_or(false)`. A to-tag-less INVITE with a stale Call-ID should take the normal initial-INVITE path; an in-dialog request whose dialog is unknown should get 481 Call/Transaction Does Not Exist (currently nothing ever returns 481 — add it for BYE/INFO/re-INVITE/REFER when dialog_for is None). handle_ack: set SipDialogStatus::Answered/Confirmed (add variant if absent). handle_bye: derive disposition from the prior status (Answered→"answered", Ringing/Routing→"no_answer").

## Digest uri parameter compared by strict string equality against rsip's re-serialized request-URI; proxy paths challenge with 401 instead of 407
Severity: low
File: /Users/loreste/palephone/src-tauri/crates/pale-server/src/sip.rs:1169 and 1147-1160
Evidence: is_authorized: `if auth.realm != realm || auth.uri != self.uri() { return false; }` — self.uri() is rsip's Uri re-serialization, which can differ textually from the bytes the client hashed and placed in the uri= parameter (explicit :5060 vs default port, parameter ordering, escaping). When they differ the request is rejected and re-challenged forever even though the password is correct; the digest itself is computed over auth.uri (sip.rs:1380, correct per RFC 2617), so the equality gate is the only thing the normalization difference breaks. Separately, digest_challenge always emits `401 Unauthorized` + WWW-Authenticate, including on proxied INVITE/MESSAGE paths where a proxy must use 407 Proxy-Authenticate / Proxy-Authorization (RFC 3261 §22.3).
Fix design: Replace the strict compare with component-wise comparison: parse both auth.uri and the request URI via rsip::Uri and compare scheme + user + lowercased host + effective port (default 5060), ignoring URI parameters; reject only on host/user mismatch. Keep using auth.uri (the client's exact string) in the HA2 computation as today. For the proxy-forwarding code paths (proxy_invite, relay_message, future relay_in_dialog_request), add a digest_proxy_challenge() variant emitting 407 with Proxy-Authenticate, and teach DigestAuth lookup to read the Proxy-Authorization header (add it to header_value/canonical_header_name); registrar/UAS paths (REGISTER, SUBSCRIBE, server-terminated INVITE) keep 401.


## 2. Architectural decision required: the registrar

VERDICT ON THE PRIOR REVIEWER'S CLAIM: verified on every substantive clause; I challenged it and found only two imprecisions. (1) Correct — the default backend (PALE_SIP_BACKEND unset => "pjsip", main.rs:383-392) builds a bare pjsua-lib UA endpoint: pjsua_create/init/start with UDP (and optional TCP/TLS) transports, zero accounts, and no module that implements registrar semantics; incoming REGISTERs are rejected by pjsua's default handler (405), since pjsua's account API only performs outbound client registration. (2) Correct — on_incoming_call (pjsip_runtime.rs:332) sends only 180 Ringing; the crate contains no 200-answer and no pjsua_call_make_call, so no call is ever routed or connected. (3) Correct in substance — on_call_media_state connects the call's conf slot to slot 0 (the sound-device master port) bidirectionally; imprecision: it's the *default* sound device, not explicitly the null device (pjsua_set_null_snd_dev is never called; null is only the headless fallback), and the callback is effectively dead code because inbound calls never reach MEDIA_ACTIVE without a 200/SDP answer. (4) Correct — two softphones cannot register to or call each other through the default backend; the pjsip backend is a telemetry shell (OPTIONS 200, dialog rows into AppState for the dashboard). (5) Correct with one correction — all registrar/proxy/PBX logic (REGISTER bindings, digest auth, INVITE 302 routing + one-shot UDP proxy, DND, call forwarding, MESSAGE relay, REFER transfer/park, SUBSCRIBE/NOTIFY) lives solely in sip.rs, which is UDP-only and refuses to start without PALE_ALLOW_INSECURE_SIP_UDP=1 (sip.rs:24-29); however it is not unauthenticated — every method enforces MD5 digest auth. Compounding bugs: main.rs spawns the parser non-fatally (SIP silently absent on gate failure while HTTP keeps serving), and login provisioning (lib.rs:917) hands every client registrar_uri=sip:{server} regardless of backend, so the shipped first-login auto-register flow fails against the default. The client itself is registrar-agnostic (register_account accepts any registrar_uri/transport; SettingsView exposes it as a free field), so third-party SIP servers work — but the documented "complete phone system, no Asterisk needed" promise (README line 7) hinges on the server registrar, which the default backend does not provide. CRITICAL severity is warranted. ARCHITECTURE RECOMMENDATION: neither (a) nor (b) as posed is the best path. Option (a) — hardening tokio sip.rs with TCP/TLS — means hand-building a SIP transaction state machine, retransmission handling, and TLS transport on top of a string parser; highest effort and risk. Option (b) — pjsua for media only — keeps a library that structurally cannot be a registrar and currently contributes nothing but transports. The honest recommendation is (c): drop pjsua-lib server-side and build the registrar/proxy as a custom module on the *lower-level pjsip endpoint API* that pjsip-sys already vendors (the pjsip-apps/samples/proxy.c pattern): this gets RFC 3261 parsing, transactions, and UDP/TCP/TLS transports from the same battle-tested stack the client uses, while reusing sip.rs's existing auth (sip_ha1 digest), location store (upsert_registration), and routing logic (DND/forwarding/302) nearly unchanged. Pragmatic alternative if FFI surface is unwanted: compose Kamailio/OpenSIPS as the registrar/proxy in docker-compose (coturn already establishes this pattern) with pale-server reduced to HTTP provisioning/API/CDR, syncing accounts via Kamailio's DB tables in the existing PostgreSQL.
